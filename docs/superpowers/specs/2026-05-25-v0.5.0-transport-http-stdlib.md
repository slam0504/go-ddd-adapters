## 1. Goal

Ship the **adapter half** of the v0.5.0 release: a public
`transport/http/stdlib` adapter that wraps Go's `net/http` server with
graceful shutdown semantics suitable for a long-running service, plus a
`transport/http/stdlib/health` sub-package that wires multiple
`ports/health.Check`s from `go-ddd-core` into `/healthz` (liveness) and
`/readyz` (readiness) endpoints.

Success criteria:

1. A downstream service can run a production-grade HTTP server +
   liveness/readiness probes by composing the adapter into a
   `bootstrap.App` with **no service-side boilerplate** for either
   server lifecycle or probe aggregation.
2. The adapter API allows running the probes on a **separate admin
   server** (different port) without code duplication, even though the
   default example wires them onto the same server.
3. `go vet`, `go build`, `go test`, `golangci-lint` (default + integration
   build tags) all green on root + `examples/orders` against a
   pseudo-version-pinned `go-ddd-core` (not yet tagged v0.5.0).

## 2. In scope

### 2.1 New packages (root module)

- `transport/http/stdlib/` (pkg name **`httpstdlib`** to avoid awkward
  `stdlib.Foo` call sites)
  - `Server` type wrapping `*http.Server` with graceful-shutdown wiring
    and a synchronously-bound listener so the actual ephemeral port
    (when `addr` is `:0`) is observable before Start returns.
  - `New(addr, handler, opts...) *Server` — primary constructor.
  - `(*Server).Module() bootstrap.ModuleFunc` — adapts the server into a
    lifecycle module.
  - `(*Server).Addr() string` — returns the actual listening address
    after `Start` (useful for `:0` ephemeral ports and tests).
  - `Module(addr, handler, opts...) bootstrap.ModuleFunc` — convenience
    wrapper equivalent to `New(addr, handler, opts...).Module()` for
    call sites that do not need `Addr()`.
  - `Option` functional-option type for `ReadHeaderTimeout`,
    `ShutdownTimeout`, `BaseContext`, `Logger`, `ModuleName`.

- `transport/http/stdlib/health/` (sub-package)
  - `Registry` type aggregating multiple `ports/health.Check`s.
  - `Handler` returning an `http.Handler` that serves both `/healthz` and
    `/readyz` under any mount point, plus per-endpoint sub-handlers
    (`LivenessHandler`, `ReadinessHandler`) for callers that want to
    mount them independently (e.g. on a separate admin server).

## 3. Out of scope (explicit follow-ups, NOT this PR)

- **`examples/orders/cmd/api` migration** from `runtime.HTTPModule` to
  `httpstdlib.Module`. The point of v0.5.0 is to land the adapter and
  prove it via its own tests; the example migration is a separate
  follow-up cycle to avoid bloating the release PR. `runtime.HTTPModule`
  stays intact this cycle (it is `cmd/internal` and not public surface).
- **Non-stdlib router flavours** (chi, gin, fiber). The package path
  `transport/http/stdlib/` is deliberately chosen so future variants live
  alongside as `transport/http/chi/` etc.
- **Custom middleware stacks** beyond what is needed for `/healthz` and
  `/readyz` themselves.
- **Adapter `go-ddd-core` v0.5.0 tag bump.** That is the gate step, not
  part of the implementation PR (see §8).
- **Changes to core's `ports/health.Check` contract.** If the contract
  is found insufficient mid-implementation, the hard rule from durable
  memory applies: push the change back into core first; do NOT grow a
  shim on the adapter side.

## 4. Non-goals

- A general-purpose HTTP framework. The adapter is a thin, opinionated
  wrapper, not a router or middleware platform.
- Backwards compatibility with `examples/orders/cmd/internal/runtime`.
  That code stays where it is; nothing depends on the new adapter
  yet.
- Replacing `bootstrap.Module` for HTTP plumbing. The adapter returns
  a `ModuleFunc`; it does not invent a new lifecycle abstraction.

## 5. Surface design

### 5.1 `httpstdlib.Server`

```go
package httpstdlib

import (
    "context"
    "net"
    "net/http"
    "time"

    "github.com/slam0504/go-ddd-core/bootstrap"
    "github.com/slam0504/go-ddd-core/ports/logger"
)

// Server wraps an *http.Server with a pre-bound listener so the actual
// listening address (post net.Listen) is exposed via Addr.
type Server struct { /* unexported */ }

// New constructs a Server. The listener is created lazily during Start
// (not at New time) so failures bind-time are reported via the
// bootstrap lifecycle, not via constructor errors.
func New(addr string, handler http.Handler, opts ...Option) *Server

// Module adapts the Server into a bootstrap.Module. Start performs
// net.Listen("tcp", configured_addr) synchronously and records the
// resolved listener address (so Addr() works), then runs Serve in a
// goroutine. Stop calls server.Shutdown with the configured timeout.
func (s *Server) Module() bootstrap.ModuleFunc

// Addr returns the actual listening address; valid only after Start
// has returned without error. Calling Addr before Start (or after a
// failed Start) returns an empty string.
func (s *Server) Addr() string

// Module is a convenience wrapper equivalent to
// New(addr, handler, opts...).Module() for callers that do not need
// Addr.
func Module(addr string, handler http.Handler, opts ...Option) bootstrap.ModuleFunc
```

**Options (functional):**

- `WithReadHeaderTimeout(d time.Duration)` — defaults to **5s** (matches
  current `runtime.HTTPModule`).
- `WithShutdownTimeout(d time.Duration)` — defaults to **15s**. Stop
  derives a `context.WithTimeout(ctx, d)` so a stuck Shutdown does not
  block the lifecycle forever.
- `WithLogger(l logger.Logger)` — defaults to a no-op logger. Used for
  start/serve-error/shutdown events.
- `WithModuleName(name string)` — defaults to `"http:" + addr`. Allows
  disambiguating multiple servers (e.g. main + admin) in logs.
- `WithBaseContext(fn func(net.Listener) context.Context)` — defaults to
  nil (stdlib default). Provided for downstream apps that need to
  propagate values into request contexts.

**Lifecycle semantics:**

- `StartFn` performs `net.Listen("tcp", configured_addr)` **synchronously**
  and stores the resolved address (so `Addr()` works for `:0`). If
  `Listen` fails — port in use, permission denied, malformed addr — the
  error is returned from `StartFn` and surfaces through the bootstrap
  lifecycle. **Only after** a successful bind does `StartFn` launch
  `server.Serve(listener)` in a goroutine and return nil.
- A non-`http.ErrServerClosed` error from `server.Serve` is logged at
  error level but does not propagate back into the lifecycle (stdlib
  pattern; the listener is already bound, so post-accept failures are
  in-flight problems, not startup problems). A future cycle could add a
  "fatal serve error" channel if needed.
- **Handler panics are not the adapter's responsibility.** `net/http`
  already recovers panics in handler goroutines (per-request) and the
  default behaviour is to close the connection and log via
  `http.Server.ErrorLog`. The adapter does not promise a `500` response
  on panic; it does not install request-level middleware. The serve-
  goroutine `defer recover()` (if any) covers only panics in the
  adapter's own serve-loop wrapper code, not user handler code.
- `StopFn` calls `server.Shutdown(timedCtx)` with `timedCtx` derived
  via `context.WithTimeout(ctx, WithShutdownTimeout)`. A timeout returns
  the underlying error (`context.DeadlineExceeded`) to the lifecycle,
  which is the right escalation for an unresponsive server.

### 5.2 `httpstdlib/health.Registry`

```go
package health

import (
    "context"
    "net/http"

    corehealth "github.com/slam0504/go-ddd-core/ports/health"
)

// Registry aggregates named health.Checks for liveness/readiness handlers.
// Zero value is ready to use; Register is safe for concurrent calls
// during startup but writes after the handler is serving should be
// avoided.
type Registry struct { /* unexported */ }

// Register adds c to the registry. Names must be unique; a duplicate
// returns an error so the wiring fails loud at startup.
func (r *Registry) Register(c corehealth.Check) error

// MustRegister is the panic-on-error variant for startup wiring.
func (r *Registry) MustRegister(c corehealth.Check)

// Handler returns an http.Handler that serves /healthz and /readyz
// via a Go 1.22 method-aware ServeMux. The handler matches the
// EXACT paths /healthz and /readyz; to mount under a prefix, wrap
// with http.StripPrefix (e.g. http.StripPrefix("/admin", r.Handler())).
// Suffix routing is intentionally rejected as too magical.
func (r *Registry) Handler() http.Handler

// LivenessHandler returns an http.Handler that serves only the liveness
// semantics (cheap 200, no Check execution).
func (r *Registry) LivenessHandler() http.Handler

// ReadinessHandler returns an http.Handler that serves only the
// readiness semantics (runs all registered Checks).
func (r *Registry) ReadinessHandler() http.Handler

// ProbeTimeout sets the per-request ctx timeout used when running
// Checks for readiness. Default 2s. A zero value disables the timeout.
func (r *Registry) ProbeTimeout(d time.Duration)
```

### 5.3 Endpoint semantics

- `/healthz` (liveness):
  - **Always returns `200 OK`** with body `{"status":"ok"}`.
  - Does **not** run any registered Check. Liveness signals "process is
    alive and able to serve HTTP" — a failing dependency must not cause
    Kubernetes / orchestrator to restart the pod.
  - Content-Type: `application/json`.

- `/readyz` (readiness):
  - Runs **every** registered Check sequentially with `ProbeTimeout`
    per Check. Sequential ordering keeps the body order stable for
    operators; concurrent execution is a future optimisation if probe
    counts grow.
  - Returns `200 OK` with body `{"status":"ok","checks":[{"name":"…","ok":true}, …]}`
    when all Checks pass.
  - Returns `503 Service Unavailable` with body
    `{"status":"unhealthy","checks":[{"name":"…","ok":false,"error":"…"}, …]}`
    when any Check fails. The body still lists every Check (passing ones
    included) so operators see partial-failure state.
  - Empty registry → `200 OK` with `{"status":"ok","checks":[]}`. (An
    empty registry is a valid "no external deps" state.)
  - Content-Type: `application/json`.

- HTTP method handling:
  - `GET` is the only supported method; everything else returns `405
    Method Not Allowed`. (Matches `net/http` `Handler` conventions and
    avoids surprising operators piping requests through generic
    health-check proxies.)

### 5.4 Separate-admin-server pattern (capability, not default)

Two-server example for an `examples/orders` follow-up, illustrated only
in package doc:

```go
reg := &health.Registry{}
reg.MustRegister(corehealth.NewCheck("postgres", pool.Ping))

app.Use(
    httpstdlib.Module(":8080", appMux, /* opts */),           // main app
    httpstdlib.Module(":9090", reg.Handler(),                 // admin only
        httpstdlib.WithModuleName("admin")),
)
```

The adapter places no constraint on whether `reg.Handler()` is mounted
on the main mux, a separate admin server, or both. This is what
"adapter allows separate admin server" means in practice.

## 6. Test plan

### 6.1 `transport/http/stdlib` tests (`*_test.go`, default build)

| Test | What it verifies |
|---|---|
| `TestServer_StartBindsListenerSynchronously` | `New(":0", h).Module().Start(...)` returns nil; `srv.Addr()` returns a non-empty `127.0.0.1:N` address; an HTTP GET against that addr returns 200. No polling needed since the listener is bound before Start returns. |
| `TestServer_StartBindErrorReturned` | A Start against an already-bound port returns the `net.Listen` error from `StartFn`; lifecycle treats it as a startup failure. |
| `TestServer_AddrEmptyBeforeStart` | `srv.Addr()` returns `""` before Start and after a failed Start. |
| `TestServer_StopGracefulShutdown` | In-flight long request completes before Stop returns; new requests after Stop fail to connect. |
| `TestServer_StopRespectsShutdownTimeout` | A hung handler causes Stop to return `context.DeadlineExceeded` after `WithShutdownTimeout` elapses. |
| `TestServer_NameAndDefaults` | Module name defaults to `"http:" + addr` and is overridable via `WithModuleName`. ReadHeaderTimeout default is 5s; ShutdownTimeout default is 15s. |
| `TestModule_ConvenienceWrapper` | The `Module(addr, handler, opts...)` package-level function produces a working module (smoke test that the wrapper composes with `New`). |

| Test | What it verifies |
|---|---|

| `TestRegistry_ReadinessAllHealthy` | All-passing Checks → 200 with each Check listed `ok:true`. |
| `TestRegistry_ReadinessOneUnhealthy` | One failing Check → 503; all Checks still listed; failing one has `ok:false` + `error:"…"`. |

### 6.3 Integration test (build tag `integration`)

One integration test combining both packages end-to-end:
`transport/http/stdlib/integration/end_to_end_test.go`

- Construct two `httpstdlib.Server`s via `New(":0", …)` — a main mux
  and an admin-only server that exposes `reg.Handler()`. Call
  `app.Use(mainSrv.Module(), adminSrv.Module())` and start the app.
- After Start, read each server's `Addr()` to drive the test HTTP
  client — no port-race polling needed because Start binds the listener
  synchronously.
- Register a passing Check (always-nil) and a flaky Check (toggled by
  test).
- HTTP-GET both probes; flip the flaky Check; HTTP-GET again; assert
  status transitions.
- Run `app.Stop` and assert both servers shut down cleanly.

(Build-tagged so Docker is not required — this is HTTP-only, no
testcontainers.)

## 7. Key invariants

1. **Adapter never imports `examples/`.** The adapter is downstream-
   consumable code; example helpers are example-only.
2. **No shim around core.** If `ports/health.Check` is found
   insufficient mid-implementation, the cycle pauses and the change is
   pushed to core first. Recorded in durable memory and in
   `.agent/decisions.md`.
3. **Liveness is dependency-free.** `/healthz` runs no Check. This is
   not a tunable.
4. **Readiness lists every Check on failure too.** Operators see
   partial-failure state; the body shape is stable.
5. **Shutdown timeout is enforced.** A stuck Shutdown surfaces as
   `context.DeadlineExceeded` to the lifecycle. Default 15s.

## 8. Tag-step gate (last action of the joint cycle)

Mirrors the durable-memory 4-step sequence (executable form of core's
"Tag at the last piece of the cycle, not the first" rule):

1. Adapter PR merged on `main`, CI green at the pseudo-version pin.
2. Core tags `v0.5.0` on its `main` (annotated, pushed).
3. Adapter `go.mod` is bumped from the pseudo-version to `v0.5.0`;
   `examples/orders/go.mod` likewise. A small follow-up bookkeeping
   commit lands on `main`.
4. Adapter CI green at the bumped tip → **then** annotate + push the
   adapter's `v0.5.0` tag.

Steps 2-4 are explicitly out of scope of the implementation PR — they
are the release ritual after merge.

## 9. Rejected alternatives

- **Combined `transport/http/` package (no `/stdlib` suffix).** Locks us
  into stdlib forever; chi/gin variants would have to live elsewhere or
  shadow names. Rejected per D2.
- **Health in the parent package, not a sub-package.** Couples the HTTP
  adapter to its health-aggregation responsibility; would force a
  separate-admin-server consumer to import the whole stdlib adapter
  just to serve probes. Rejected.
- **Liveness runs Checks too.** Causes orchestrators to restart pods on
  transient dep failures — anti-pattern. Rejected.
- **Empty registry → 503 readiness.** Forces every service to register
  at least one Check even when none apply (e.g. stateless transforms).
  Rejected; empty → 200.
- **Concurrent Check execution by default.** Premature complexity at
  current probe counts; ordering would also be non-deterministic in
  the response body. Deferred until a real use case shows up.
